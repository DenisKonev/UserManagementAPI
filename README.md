- ### Дизайн таблиц в реляционной БД
### 1. Хранение всей информации в одной таблице

**Преимущества:**

- **Простота:** Все данные пользователя находятся в одном месте, что упрощает запросы и управление.
- **Производительность:** Меньше соединений (JOIN) при выборке данных, что может повысить скорость выполнения запросов.
- **Удобство:** Легче реализовать базовые операции CRUD (Create, Read, Update, Delete).

**Недостатки:**

- **Избыточность:** Если некоторые данные редко используются или могут изменяться независимо, хранение их вместе может привести к избыточности.
- **Гибкость:** Менее гибкая структура при необходимости расширения функциональности в будущем.

### 2. Разделение информации на несколько таблиц

**Возможные подходы:**

- **Основная таблица `Users` и отдельная таблица для фотографий `UserPhotos`:**

  **Таблица `Users`:**
    - `user_id` (PK)
    - `surname`
    - `name`
    - `middle_name`
    - `date_of_birth`
    - `email`
    - `phone_number`    

  **Таблица `UserPhotos`:**  
    - `photo_id` (PK)
    - `user_id` (FK)
    - `photo` (BLOB или ссылка на хранилище файлов)

**Преимущества:**

- **Разделение ответственности:** Отделение больших или редко используемых данных (например, фотографий) может улучшить производительность основной таблицы.
- **Гибкость:** Легче управлять и расширять отдельные аспекты данных.
- **Безопасность:** Можно применять разные уровни доступа к различным таблицам.

**Недостатки:**

- **Сложность:** Необходимость выполнения дополнительных JOIN-запросов при выборке полной информации о пользователе.
- **Управление целостностью:** Требует дополнительных мер для обеспечения согласованности данных между таблицами.

>***Было выбрано решение с одной таблицей, так как данных в БД мало, чтобы основать второй подход***
